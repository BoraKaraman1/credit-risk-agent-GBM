# Go Services

Go port of the serving and monitoring layer: the scoring API, the
monitoring agents, and the Supabase sync job. Model **training** stays
in Python (LightGBM) — the Go retrain orchestrator shells out to it.

## What runs where

| Component | Language | Why |
|-----------|----------|-----|
| `cmd/scoring-api` | Go | `/score`, `/score/batch`, `/health`, `/reload` — pure-Go inference + TreeSHAP |
| `cmd/drift-monitor` | Go | PSI on scores, CSI per feature |
| `cmd/performance-monitor` | Go | AUC / KS / Gini, decile rank-ordering |
| `cmd/retrain-orchestrator` | Go | Orchestrates retraining; calls `pipeline/train_challenger.py` for the LightGBM fit, evaluates champion vs challenger natively |
| `cmd/supabase-sync` | Go | Bulk upsert of Gold features + training score distribution |
| `pipeline/` (bronze/silver/gold, training, fairness, GX) | Python | LightGBM / SHAP / Great Expectations have no Go equivalents |
| `dags/` | Python | Airflow |

## How inference works without Python

`pipeline/export_model_json.py` dumps the champion LightGBM model
(tree nodes, baseline, metadata) to `data/models/champion/model.json`,
normalizing LightGBM's tree dump into a library-agnostic columnar
format (`value <= threshold` goes left, NaN follows
`missing_go_to_left`). `internal/model` loads it and implements:

- **`predict_proba`** — tree traversal verified to 1e-9 against
  LightGBM's own predictions.
- **TreeSHAP** — path-dependent algorithm (Lundberg et al., Algorithm 2)
  for the ECOA adverse action reasons, verified to 1e-6 against the
  Python `shap` library, including rows with missing values.

Re-export whenever the champion changes (the retrain flow exports the
challenger automatically):

```bash
.venv/bin/python pipeline/export_model_json.py            # champion
.venv/bin/python pipeline/export_model_json.py data/models/challenger
```

## Running

All binaries read `.env` (`DATABASE_URL`) and expect to run from the
repository root (or set `CREDIT_RISK_DATA_DIR` / `CREDIT_RISK_MODELS_DIR`).

```bash
cd go
go build -o bin/ ./cmd/...
cd ..

PORT=8000 ./go/bin/scoring-api      # POST /score {"applicant_id": "LC_0000001"}
./go/bin/drift-monitor              # JSON report on stdout, logs to drift_log
./go/bin/performance-monitor
./go/bin/retrain-orchestrator psi_critical   # reason arg, default "manual"
./go/bin/supabase-sync
```

The monitors fall back to the Gold test set as a production proxy when
`DATABASE_URL` is unset or `scoring_log` is empty, exactly like the
Python agents. `PYTHON_BIN` overrides the interpreter used for
challenger training (default `.venv/bin/python`, then `python3`).

## Tests

```bash
cd go && go test ./...
```

The test suites cross-check against fixtures generated by the Python
stack (`internal/model/testdata`, `internal/metrics/testdata`):
LightGBM `predict_proba`, `shap` TreeSHAP values, numpy PSI/CSI,
sklearn AUC/KS, and pandas decile analysis.
